## Story

**Como** usuário do portal Lizard Web,
**Quero** acessar uma lista paginada de OPs com suporte a filtros por data, classe e busca por número,
**Para que** o frontend possa exibir e navegar pelo histórico de operações de forma eficiente sem carregar todos os dados de uma vez.

---

## Acceptance Criteria

- [x] AC-01: `GET /ops` retorna JSON com campos `items`, `total`, `page`, `limit`, `pages`
- [x] AC-02: `items` é lista de objetos com `op`, `last_updated`, `total_hides`, `avg_yield`, `predominant_class`
- [x] AC-03: Suporta query params `page` (default 1) e `limit` (default 20, max 100)
- [x] AC-04: Suporta filtro `date_from` e `date_to` (formato `YYYY-MM-DD`) aplicados sobre `date(GrdDat)`
- [x] AC-05: Suporta filtro `class` aplicado sobre `GrdCla` (case-insensitive)
- [x] AC-06: Suporta filtro `search` aplicado sobre `GrdOp` como LIKE `%search%`
- [x] AC-07: Paginação correta: `pages = ceil(total / limit)`, `items` contém apenas os registros da página
- [x] AC-08: Toda query usa parameterized statements — sem concatenação de strings em SQL
- [x] AC-09: Endpoint retorna 200 com `items: []` e `total: 0` quando não há dados (nunca 500)
- [x] AC-10: `lizard-web/api/routers/ops.py` não importa nenhum módulo de `lizard/`
- [x] AC-11: `limit` é validado — valores > 100 retornam 422 com mensagem clara

**Response:**

```json
      "predominant_class": "A"
    }
  ],
  "total": 142,
  "page": 1,
  "limit": 20,
  "pages": 8
}
```

### Mapeamento SQLite → API
[Source: docs/architecture/system-architecture.md#5-mapeamento-de-dados-sqlite-api]

| Campo API | Coluna SQLite |
|-----------|--------------|
| `op` | `GrdOp` |
| `last_updated` | `MAX(GrdDat)` |
| `total_hides` | `COUNT(*)` |
| `avg_yield` | `AVG(GrdUsaPer)` |
| `predominant_class` | moda de `GrdCla` (subquery) |

### Query Base de Referência

```python
# Query base (sem filtros) — adaptar com filtros dinâmicos via params
cursor.execute("""
    SELECT GrdOp as op,
           MAX(GrdDat) as last_updated,
           COUNT(*) as total_hides,
           ROUND(AVG(GrdUsaPer), 2) as avg_yield,
           (
               SELECT GrdCla FROM grading_results g2
               WHERE g2.GrdOp = g1.GrdOp
               GROUP BY GrdCla
               ORDER BY COUNT(*) DESC
               LIMIT 1
           ) as predominant_class
    FROM grading_results g1
    WHERE 1=1
      -- filtros opcionais via params:
      -- AND date(GrdDat) >= ?         (date_from)
      -- AND date(GrdDat) <= ?         (date_to)
      -- AND UPPER(GrdCla) = UPPER(?)  (class)
      -- AND GrdOp LIKE ?              (search: '%' + search + '%')
    GROUP BY GrdOp
    ORDER BY last_updated DESC
    LIMIT ? OFFSET ?
""", params)
```

### Estratégia de Filtros Dinâmicos

Construir `WHERE` e `params` de forma segura:

```python
conditions = ["1=1"]
params = []

if date_from:
    conditions.append("date(GrdDat) >= ?")
    params.append(date_from)
if date_to:
    conditions.append("date(GrdDat) <= ?")
    params.append(date_to)
if class_filter:
    conditions.append("UPPER(GrdCla) = UPPER(?)")
    params.append(class_filter)
if search:
    conditions.append("GrdOp LIKE ?")
    params.append(f"%{search}%")

where_clause = " AND ".join(conditions)
```

**NUNCA** interpolar strings na query — usar apenas `?` com lista de parâmetros.

### Paginação

```python
import math

offset = (page - 1) * limit
pages = math.ceil(total / limit) if total > 0 else 0
```

### Validação de Parâmetros com FastAPI

```python
from fastapi import APIRouter, Query
from typing import Optional

@router.get("/ops", response_model=OpListResponse)
def list_ops(
    page: int = Query(default=1, ge=1),
    limit: int = Query(default=20, ge=1, le=100),
    date_from: Optional[str] = Query(default=None),
    date_to: Optional[str] = Query(default=None),
    class_filter: Optional[str] = Query(default=None, alias="class"),
    search: Optional[str] = Query(default=None),
):
```

`le=100` em `limit` faz o FastAPI retornar 422 automaticamente para valores > 100 (AC-11).

### Segurança
[Source: docs/architecture/system-architecture.md#6-segurança]

- `search` param: `f"%{search}%"` é safe pois vai como parâmetro `?`, não interpolado na query
